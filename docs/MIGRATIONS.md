# 🚀 Система миграций и критических патчей Reshala

Этот документ описывает работу «Движка миграций» (Migration Engine), который позволяет автоматически предлагать пользователю важные изменения конфигурации и патчи безопасности сразу в главном меню.

---

## 1. Зачем это нужно?
В отличие от обычного обновления кода (которое меняет скрипты), миграции меняют **состояние системы** (конфиги, правила файрвола, пакеты). Система миграций позволяет:
1.  **Не заставлять** пользователя вручную перенастраивать модули после обновлений.
2.  **Гарантировать**, что критические исправления (например, дыры в безопасности Docker) будут замечены и применены.

---

## 2. Как устроен «Движок» (Engine)
Движок находится в `modules/utils/migrations.sh`. Его задачи:
-   Сканировать папку `modules/utils/migrations/` на наличие `.sh` файлов.
-   Загружать их по порядку (0001, 0002...).
-   Проверять, применен ли патч (по логу в `/etc/reshala/applied_migrations.log`).
-   Выводить уведомление в главном меню, если есть «хвосты».

---

## 3. Как создать новую миграцию (Reshala Patch Standard)

### Шаг 1: Создание файла
Создайте новый файл в `modules/utils/migrations/` с порядковым номером:
`0002_fix_ssh_limits.sh`

### Шаг 2: Оформление шапки
Каждый файл миграции **обязан** иметь стандартный заголовок:

```bash
#!/bin/bash
# 000x_description.sh - Краткое название
#
# ══════════════════════════════════════════════════════════════════════════
# 🛠 ШАБЛОН МИГРАЦИИ (Reshala Patch Standard)
# ══════════════════════════════════════════════════════════════════════════
# Каждая миграция — это независимый модуль, который загружается движком.
# Движок вызывает migration_check() для проверки и migration_apply() для дела.
# ══════════════════════════════════════════════════════════════════════════

MIG_ID="UNIQUE_ID_2024"  # Уникальный ID для лога
MIG_TITLE="Заголовок"    # То, что увидит юзер в списке
MIG_DESC="Описание"      # Подробности под заголовком
```

### Шаг 3: Реализация логики
Вы должны реализовать две функции:

1.  **`migration_check()`**:
    -   Должна вернуть `0`, если патч **НУЖЕН**.
    -   Должна вернуть `1`, если патч уже стоит или не применим.
2.  **`migration_apply()`**:
    -   Логика «лечения» системы.
    -   Возвращает `0` при успехе.

---

## 4. Жизненный цикл миграции
1.  Пользователь запускает `reshala.sh`.
2.  Движок видит новый файл `0002_...`.
3.  `migration_check` возвращает `0`.
4.  В главном меню горит красный баннер: `🔥 ЕСТЬ КРИТИЧЕСКИЕ ОБНОВЛЕНИЯ БЕЗОПАСНОСТИ`.
5.  Пользователь нажимает `m`, видит описание патча и подтверждает его.
6.  `migration_apply` отрабатывает.
7.  Движок записывает `MIG_ID` в `/etc/reshala/applied_migrations.log`.
8.  Баннер исчезает.

---

## 5. Правила хорошего тона
-   Миграции должны быть **идемпотентными** (безопасными при повторном запуске).
-   Используйте `is_migration_applied "$MIG_ID"` внутри `migration_check`.
-   Если миграция требует другого модуля (например, `firewall.sh`), движок сам его подгрузит (если вы укажете это в функции), но лучше делать проверки на существование команд.
